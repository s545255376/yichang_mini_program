## 版本号控制规范 2022-07-04 by 吴枫

### 前言

自**诚美匠选**的前身**去哪美拼购小程序**起，这个app（后称此项目）的版本控制一直延续使用**npm version**的「x.y.z版本号」控制规范，此项目版本号控制规范受限于微信开发者工具内置的「版本管理」功能，所以版本号规范不借助三方工具，使用纯手写方式编写版本号，因此要求使用**英文数字、英文符号**进行规范化编写。

### 版本号控制规范 **x.x.xxx**

我们使用 **major** | **minor** | **patch** 来控制版本位

- **major**: 主版本位，当系统发生巨大变更，重新设计营销体系，如去哪美拼购1.0迁移匠选小程序，不论从前后台的逻辑规范，到营销方案、玩法的变化都发生巨大变更，此时可使用主版本位进行进位更新。
- **minor**: 次版本位，当系统发生较大改变，并不影响旧代码的兼容性，或很小影响兼容性，不会对客户使用习惯造成极大影响的，如分包的删除或增加，主页tabBar发生增减或变更，主包页面组大量的增加或者大范围的减少等，此时可使用次版本位进行进位更新。
- **patch**: 补丁版本位，因此项目依托于公司营销部门的需求和业务进行技术支持，所以敏捷开发会大量使用此版本位进行更新或bug的修复，目前此版本位可使用**最多3位数**代替补丁更新，此版本位较活，可根据开发规模自行决定。


## Git编写和提交规范 2022-07-04 by 吴枫

### 前言

本项目使用[约定式提交规范](https://www.conventionalcommits.org/zh-hans/v1.0.0) 来规范提交信息，方便利用一组简单的轻量级约定来创建清晰的提交历史，一方面更有利于前端技术团队后续尝试编写自动化工具打下基础，另一方面也可提升团队效率。

### 提交说明

和版本控制一样，由于使用微信开发者工具进行代码日常管理和版本提交，提交记录不能使用工具进行强制规范，虽然很想加入一些强制化规范工具，但目前暂定纯手写规范，后续可能会尝试使用三方工具来规范提交历史，因此**提交格式**需要相当注意。

### 提交格式

> <类型:必填><范围:可选><英文冒号:必填><空格:必填><简单描述:必填>

可以按如下提交：

```
feat(app): 修改了app.js微信获取设备信息api的执行方法
```

或

```
feat: 修改了app.js微信获取设备信息api的执行方法
```


#### 结构化元素

| 类型   | 含义 |
| :----- | :----- |
| feat |  新增功能  | 
| fix |  	修复 bug  | 
| docs |  	更改文档  | 
| style |  不影响代码含义的变化(空白，格式化，缺少分号等)  | 
| refactor |  重构，不修复 bug 且不添加功能  | 
| perf |  性能优化代码  | 
| build |  构建方面相关， 如新增modules包等  |
